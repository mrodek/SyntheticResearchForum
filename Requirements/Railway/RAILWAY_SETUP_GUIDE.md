# SRF Railway Setup Guide

Step-by-step instructions for configuring the Railway OpenClaw service to run the Synthetic Research Forum pipeline.

## Architecture Overview

SRF uses a two-repo deployment model:

- **`mrodek/clawdbot-railway-template`** (forked from `vignesh07/clawdbot-railway-template`) — the Railway service source. Builds and runs the OpenClaw Gateway. This is what Railway deploys.
- **`mrodek/SyntheticResearchForum`** — the SRF codebase. Cloned onto the persistent `/data` volume at startup via a bootstrap hook. OpenClaw calls SRF Python scripts from `/data/srf/`.

The template's runtime image already includes Python + venv support. SRF Python packages are installed into a persistent venv at `/data/venv` via `/data/workspace/bootstrap.sh`, which the template wrapper runs automatically on every startup.

## Prerequisites

- Railway account
- GitHub account with:
  - `mrodek/SyntheticResearchForum` repo (SRF codebase)
  - `mrodek/clawdbot-railway-template` fork (Railway service source — see Step 1)
- Anthropic (or OpenAI) API key

---

## Step 1 — Fork the OpenClaw Template

Railway can only connect repos you own. You must fork the template before connecting it.

1. Go to `github.com/vignesh07/clawdbot-railway-template`
2. Click **Fork** → fork to your account (`mrodek`)
3. The fork appears as `mrodek/clawdbot-railway-template`

---

## Step 2 — Connect the Fork to Railway

1. Railway → your OpenClaw service → **Settings** tab
2. Under **Source** → click **Connect Repo**
3. Select `mrodek/clawdbot-railway-template`
4. Branch: `main`

Railway builds from the template's own `Dockerfile`, which compiles OpenClaw from source. Do not point Railway at the SRF repo — the SRF code arrives via the bootstrap hook (Step 7).

---

## Step 3 — Add a Volume

Workspace artifacts, OpenClaw state, the SRF code clone, and the Python venv all live on a persistent volume. Without it, everything is lost on restart.

1. Railway → service → **Volumes** tab → **Add Volume**
2. Mount path: `/data`

---

## Step 4 — Enable HTTP Proxy

1. Railway → service → **Settings** → **Networking** → **Add HTTP Proxy**
2. Change the target port from `3000` to `8080`

Railway assigns a public URL: `https://<something>.up.railway.app`

---

## Step 5 — Set Environment Variables

Railway → service → **Variables** tab.

### Required — service will not start without these

| Variable | Value |
|----------|-------|
| `SETUP_PASSWORD` | A strong password (protects `/setup` and `/openclaw` via HTTP Basic auth) |
| `PORT` | `8080` |
| `SRF_LLM_PROVIDER` | `anthropic` (or `openai`) |
| `SRF_LLM_MODEL` | `claude-haiku-4-5-20251001` (cheap model for integration testing; switch to `claude-sonnet-4-6` for production) |
| `SRF_LLM_API_KEY` | Your Anthropic / OpenAI API key |

### Required for persistent state

| Variable | Value |
|----------|-------|
| `OPENCLAW_STATE_DIR` | `/data/.openclaw` |
| `OPENCLAW_WORKSPACE_DIR` | `/data/workspace` |

### Recommended — secures MCP and webhook endpoints

| Variable | Value |
|----------|-------|
| `OPENCLAW_GATEWAY_TOKEN` | Pre-generated by template on first deploy — leave as-is, or generate: `openssl rand -hex 32` |

### Optional — PromptLedger observability (add later)

Both must be set together, or both omitted. System runs without observability if absent.

| Variable | Value |
|----------|-------|
| `PROMPTLEDGER_API_URL` | Your PromptLedger instance URL |
| `PROMPTLEDGER_API_KEY` | Project-scoped key from `POST /v1/admin/projects` |

### Optional runtime tuning (defaults are fine for initial testing)

| Variable | Default | Description |
|----------|---------|-------------|
| `SRF_LOG_LEVEL` | `INFO` | `DEBUG`, `INFO`, `WARNING`, or `ERROR` |
| `SRF_ARXIV_DELAY_SECONDS` | `3` | Politeness delay between arXiv fetches |
| `SRF_MIN_PAPERS` | `2` | Minimum papers required to run a forum |
| `SRF_MIN_AGENTS` | `2` | Minimum Paper Agents required for debate |
| `SRF_MAX_PREP_RETRIES` | `3` | LLM call retries during agent preparation |

---

## Step 6 — Deploy and Confirm Green

After setting variables, Railway will auto-deploy. The build compiles OpenClaw from source — expect 2–4 minutes.

A healthy startup in the logs looks like:
```
OpenClaw Gateway listening on :8080
```

Then `/health` should return 200.

### Common failures

| Symptom | Fix |
|---------|-----|
| `502 Bad Gateway` | Volume not mounted at `/data`, or `OPENCLAW_STATE_DIR`/`OPENCLAW_WORKSPACE_DIR` not set |
| `Gateway not ready` | Check logs for the reason — usually missing volume or bad env vars |
| Build OOM | Upgrade Railway plan to 2GB+ memory (building OpenClaw from source is memory-intensive) |
| `disconnected (1008): pairing required` | Normal — gateway is running but no device approved yet. Fix via `/setup` wizard |

## Step 7 — Run the /setup Wizard

```
https://<your-service>.up.railway.app/setup
```

Your browser will prompt for **HTTP Basic auth** — use any username, password is `SETUP_PASSWORD`.

1. Choose provider: **Anthropic**, paste your API key
2. Skip Telegram / Discord (not needed for SRF)
3. Click **Run setup**

---

## Step 8 — Create the Bootstrap Script

The template wrapper runs `/data/workspace/bootstrap.sh` on every startup if it exists. This is how SRF Python code gets installed onto the service.

Use the OpenClaw exec tool (at `/openclaw`) to create the file:

```bash
cat > /data/workspace/bootstrap.sh << 'EOF'
#!/usr/bin/env bash
set -euo pipefail

# Clone or update SRF repo onto the persistent volume
if [ ! -d /data/srf ]; then
  git clone https://github.com/mrodek/SyntheticResearchForum /data/srf
else
  cd /data/srf && git pull --ff-only
fi

# Install SRF into a persistent venv (survives redeploys)
python3 -m venv /data/venv 2>/dev/null || true
/data/venv/bin/pip install -e /data/srf[anthropic,openai]

# Copy skills to workspace so OpenClaw discovers them on next startup
mkdir -p /data/workspace/skills
cp -rn /data/srf/skills/. /data/workspace/skills/ 2>/dev/null || true
EOF
chmod +x /data/workspace/bootstrap.sh
```

Then trigger a redeploy so the bootstrap script runs. After the redeploy:
- `/data/srf/` — SRF codebase
- `/data/venv/` — Python venv with `srf`, `anthropic`, `openai` installed
- `/data/workspace/skills/` — SRF OpenClaw skills

---

## Step 9 — Run srf_init.py

Via the OpenClaw exec tool:

```bash
/data/venv/bin/python /data/srf/scripts/srf_init.py
```

Expected output:
```
SRF init complete
```

This confirms:
- Required environment variables are present
- Workspace directories created under `/data/workspace/`
- `srf` Python package is importable from the venv
- PromptLedger prompts registered (if configured) or gracefully skipped

## Step 10 — First Integration Test

Test the pre-debate pipeline via the exec tool:

```bash
/data/venv/bin/python /data/srf/scripts/parse_newsletter.py \
  --source-path /path/to/newsletter.md \
  --output-dir /data/workspace/newsletters
```

| Step | Command | Expected output |
|------|---------|-----------------|
| Workspace setup | `python /data/srf/scripts/run_workspace_setup.py` | Forum workspace at `/data/workspace/forum/{id}/` |
| Paper extraction | `python /data/srf/scripts/run_paper_extraction.py` | PDFs fetched, text extracted |
| Agent preparation | `python /data/srf/scripts/run_preparation.py` | `PreparationArtifact` files in `preparation/` |
| Debate | — | Not yet implemented (Epic 6) |

All commands use `/data/venv/bin/python` and absolute paths to `/data/srf/scripts/`.

## Connecting Claude Desktop (MCP)

Once the service is running and `/setup` is complete:

1. Open Claude Desktop → Settings → Developer → Edit Config
2. Add the OpenClaw MCP server:

```json
{
  "mcpServers": {
    "srf": {
      "url": "https://<your-service>.up.railway.app/mcp",
      "headers": {
        "Authorization": "Bearer <OPENCLAW_GATEWAY_TOKEN>"
      }
    }
  }
}
```

3. Restart Claude Desktop
4. The `trigger_newsletter_forum`, `review_forum_debate_format`, and `approve_editorial_review` skills will appear as tools

> **Note:** The skills currently reference `python scripts/...` with relative paths. Once the bootstrap approach is confirmed working, the SKILL.md files need updating to use `/data/venv/bin/python /data/srf/scripts/...`.
